# Описание микросервиса File Service

## Общее описание

**File Service** — микросервис для хранения и управления файлами в распределённой системе. Сервис предоставляет REST API для загрузки, хранения и получения файлов. Реализован на языке Go с использованием чистой архитектуры и разделением на слои.

### Основные функции:
- Загрузка файлов с автоматическим определением типа (изображения/документы)
- Хранение метаданных файлов в PostgreSQL
- Хранение файлов в объектном хранилище MinIO
- Получение информации о файлах по идентификатору
- Пакетное получение файлов по нескольким идентификаторам

### Технологический стек:
- **Язык программирования:** Go
- **База данных:** PostgreSQL (метаданные файлов)
- **Объектное хранилище:** MinIO (S3-совместимое хранилище)
- **HTTP-сервер:** Собственная библиотека gohttp/goserver
- **Миграции:** goose

---

## Архитектура

Микросервис построен по принципам **чистой архитектуры** с разделением на следующие слои:

### 1. Слой представления (API Layer)
- **Пакет:** `api/`, `api/handler/`
- **Компоненты:**
  - `ServerBuilder` — построитель HTTP-сервера с поддержкой middleware (метрики, логирование, recovery)
  - `FileHandler` — обработчики HTTP-запросов для работы с файлами

### 2. Слой бизнес-логики (Service Layer)
- **Пакет:** `service/file/`
- **Компоненты:**
  - `Service` — основная бизнес-логика управления файлами
  - `File` — доменная модель файла
  - `Bucket` — перечисление типов хранилищ (images, documents)
  - `Storage` — интерфейс объектного хранилища
  - `Repository` — интерфейс репозитория метаданных

### 3. Слой данных (Data Layer)
- **Пакет:** `database/file/`, `database/object/`
- **Компоненты:**
  - `Repository` — реализация репозитория для PostgreSQL
  - `Minio` — реализация хранилища для MinIO
  - `Mappers` — маппинг между моделями базы данных и доменными моделями

### 4. Слой конфигурации (Config Layer)
- **Пакет:** `config/`
- **Компоненты:**
  - `Settings` — структура настроек приложения
  - `Databases` — конфигурация подключений к базам данных

### 5. Слой приложения (Application Layer)
- **Файл:** `app.go`
- **Компоненты:**
  - `App` — главный класс приложения, управляющий жизненным циклом сервиса

## API Reference

### Таблица эндпоинтов

| URL | Описание | Параметры | Тип запроса |
|-----|----------|-----------|-------------|
| `/files` | Загрузка нового файла | `File` — файл в формате multipart/form-data | POST |
| `/files/{id}` | Получение информации о файле по ID | `id` — идентификатор файла (path parameter, int) | GET |
| `/files?id=...` | Получение информации о нескольких файлах | `id` — идентификаторы файлов (query parameter, int[], можно указать несколько раз) | GET |

### Описание эндпоинтов

#### POST /files — Загрузка файла

**Описание:** Загружает файл в хранилище. Тип хранилища (bucket) определяется автоматически по расширению файла:
- Изображения (`.jpg`, `.jpeg`, `.png`) → bucket `images`
- Остальные файлы → bucket `documents`

**Формат запроса:** `multipart/form-data`

**Параметры:**
| Параметр | Тип | Обязательный | Описание |
|----------|-----|--------------|----------|
| File | file | Да | Файл для загрузки |

**Пример ответа:**
```json
{
    "ID": 1,
    "FileName": "document.pdf",
    "FileSize": 102400,
    "Bucket": "documents",
    "URL": "http://minio:9000/documents/document.pdf"
}
```

---

#### GET /files/{id} — Получение файла по ID

**Описание:** Возвращает метаданные файла по его идентификатору.

**Параметры:**
| Параметр | Тип | Расположение | Обязательный | Описание |
|----------|-----|--------------|--------------|----------|
| id | int | path | Да | Идентификатор файла |

**Пример запроса:**
```
GET /files/1
```

**Пример ответа:**
```json
{
    "ID": 1,
    "FileName": "image.jpg",
    "FileSize": 51200,
    "Bucket": "images",
    "URL": "http://minio:9000/images/image.jpg"
}
```

---

#### GET /files?id=... — Получение нескольких файлов

**Описание:** Возвращает метаданные нескольких файлов по их идентификаторам.

**Параметры:**
| Параметр | Тип | Расположение | Обязательный | Описание |
|----------|-----|--------------|--------------|----------|
| id | int[] | query | Да | Идентификаторы файлов (можно указать несколько раз) |

**Пример запроса:**
```
GET /files?id=1&id=2&id=3
```

**Пример ответа:**
```json
[
    {
        "ID": 1,
        "FileName": "image.jpg",
        "FileSize": 51200,
        "Bucket": "images",
        "URL": "http://minio:9000/images/image.jpg"
    },
    {
        "ID": 2,
        "FileName": "document.pdf",
        "FileSize": 102400,
        "Bucket": "documents",
        "URL": "http://minio:9000/documents/document.pdf"
    }
]
```

## Модель данных

### Таблица storage_files

| Поле | Тип | Описание |
|------|-----|----------|
| id | int (PK, auto) | Уникальный идентификатор файла |
| file_name | text | Имя файла |
| file_size | int | Размер файла в байтах |
| bucket | text | Название хранилища (images/documents) |
| url | text | Ссылка для скачивания файла |

## Диаграмма классов

Ниже представлена диаграмма классов микросервиса, созданная с помощью PlantUML.

**Расположение файла диаграммы:** `doc/class-diagram.puml`

### Описание диаграммы классов

Диаграмма классов отображает основные компоненты микросервиса и их взаимосвязи:

1. **App** — центральный класс приложения, который управляет жизненным циклом сервиса. Содержит ссылки на все основные компоненты: конфигурацию, HTTP-сервер, подключения к базам данных и сервисы.

2. **ServerBuilder** — построитель HTTP-сервера, реализующий паттерн Builder. Отвечает за конфигурацию маршрутов и middleware.

3. **FileHandler** — содержит обработчики HTTP-запросов. Использует вспомогательные структуры `fileIDVars` и `fileIDsVars` для парсинга параметров запросов.

4. **Service** — основной класс бизнес-логики. Реализует операции загрузки и получения файлов. Зависит от интерфейсов `Storage` и `Repository`, что обеспечивает слабую связанность компонентов.

5. **File** (доменная модель) — представляет файл в бизнес-логике с полями: ID, FileName, FileSize, Bucket, URL.

6. **Bucket** — перечисление типов хранилищ: `images` для изображений и `documents` для остальных файлов.

7. **Storage** (интерфейс) — абстракция объектного хранилища. Реализован классом `Minio`.

8. **Repository** (интерфейс) — абстракция репозитория метаданных. Реализован классом `DBRepository`.

9. **Minio** — реализация интерфейса `Storage` для работы с MinIO. Обеспечивает загрузку файлов и формирование URL для доступа.

10. **DBRepository** — реализация интерфейса `Repository` для работы с PostgreSQL. Использует `DBFile` как модель базы данных и `Mappers` для преобразования между слоями.

11. **Settings**, **Databases**, **MinioConfig** — классы конфигурации, загружаемые из JSON-файлов и переменных окружения.

### Ключевые паттерны проектирования:
- **Dependency Injection** — зависимости инжектируются через конструкторы
- **Repository Pattern** — абстракция доступа к данным
- **Builder Pattern** — построение HTTP-сервера
- **Strategy Pattern** — выбор хранилища по типу файла

## Конфигурация

Сервис поддерживает конфигурацию через JSON-файлы и переменные окружения:

### Переменные окружения:
| Переменная | Описание |
|------------|----------|
| MINIO_USER | Пользователь MinIO |
| MINIO_PASSWORD | Пароль MinIO |
| POSTGRES_PASSWORD | Пароль PostgreSQL |
| ENVIRONMENT | Окружение (local/dev/prod) |

### Структура конфигурационного файла:
```json
{
    "port": 8080,
    "databases": {
        "postgres": "postgres://user:%s@host:5432/database?sslmode=disable",
        "minio": {
            "endpoint": "minio:9000",
            "use_ssl": false,
            "host": "http://minio:9000"
        }
    }
}
```
